Overview

RoboJournal is capable of receiving additional arguments prior to launch. These arguments are meant to be manually entered at a command line (i.e. Bash prompt on Linux/Unix or cmd.exe on Windows) or through a program launch window (such as the Run Command on Windows).

Invoking arguments is inherently easier on Linux because command prompts are far more accessible. For instance, pressing the ALT and F2 keys simultaneously displays a "Run Program" dialog on common desktop environments like KDE or GNOME/MATE. On Windows XP and older, the "Run" option on the Start Menu was the easiest way to launch a program with custom arguments. On Windows 7 and later, the Run Command still exists but is hidden by default. As such, you must re-enable (Windows 8 method here) the Run Command before you can launch RoboJournal with custom arguments.

Date Override

Under normal conditions, RoboJournal applies the current date to all new entries created during that session. However, you can force RoboJournal 0.5 (and later) to use an alternate date for new entries with a hidden but useful function that is activated prior to the current session. This function is universally referred to as the "date override" in this documentation.

The date override allows you to set a false date of origin for entries written during a session in which date override is active. For instance, you can use the date override to make it appear as though an entry was written on 12/31/2013 even if you actually wrote it on 1/1/2014. This can be useful in situations where your duties require you to keep a daily log. If you forget to write an entry for one or more days, you can use the date override to "fill in" the missing entries at a later time. Date override does not affect the dates of entries that are edited during the session.

FYI: The date override is convenient in situations where it is necessary but it can also corrupt the entire database when used incorrectly. The date override should only be applied to a new entry that will be the most recent item in the timeline once it has been saved. Never use the date override to insert a new entry between other entries at a previous point in the timeline or at a future date (doing either action always causes database corruption).

Proper Syntax

You can trigger the date override feature by appending --date-override=mm-dd-yyyy to the RoboJournal executable path (for instance, C:\Program Files (x86)\RoboJournal\robojournal.exe --date-override=mm-dd-yyyy). The mm-dd-yyyy portion of the argument refers to the date you want to use (in mm/dd/yyyy format) instead of the current date. For instance, you could use --date-override=01-01-2013 to set the current session's date to January 1, 2013.

RoboJournal accepts and initiates the override if the specified date is valid; for instance, 02-28-2013 is acceptable while 02-30-2013 is not because February 30 does not exist in any year. RoboJournal ignores the date override argument and operates normally if the specified date is invalid and/or the argument syntax is incorrect. You can separate the month, day, and year values in the date with hyphens or forward slashes (but not a mix of both). Enabling Date Override on Windows

Windows does not accept the --date-override argument when the program is launched through a desktop shortcut. The only way to initiate date override from Windows is through the Run Command window on the Windows Start menu:Figure 2: Under the current settings, RoboJournal's date override is set for September 3, 2013.

FYI: Windows usually puts quotation marks around commands in the Open field, especially if they were added with the Browse button. You must remove the quotation marks for the date override command to be valid (i.e. C:\Program Files (x86)\RoboJournal\robojournal.exe --date-override=09-03-2013 is correct while "C:\Program Files (x86)\RoboJournal\robojournal.exe --date-override=09-03-2013" is not).
